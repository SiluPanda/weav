Getting Started

Weav is an in-memory context graph database for AI, written in Rust. It combines graph storage, vector search, and budget-aware context retrieval into a single binary with sub-millisecond latency. Designed for AI agents, GraphRAG pipelines, and LLM applications.

Weav exposes three protocol interfaces simultaneously:

ProtocolPortUse Case
RESP36380Redis-compatible wire protocol for CLI and existing tooling
gRPC6381High-performance binary protocol for service-to-service
HTTP6382REST API for SDKs, curl, and web applications

Quickstart

Start the server with default settings, create a graph, add some data, and query context. The examples below use curl and the Python SDK side by side.

# Start the server
cargo run --release -p weav-server

# Create a graph
curl -X POST http://localhost:6382/v1/graphs \
  -H "Content-Type: application/json" \
  -d '{"name": "knowledge"}'

# Add nodes
curl -X POST http://localhost:6382/v1/graphs/knowledge/nodes \
  -H "Content-Type: application/json" \
  -d '{
    "label": "Person",
    "properties": {"name": "Alice", "role": "Engineer"},
    "entity_key": "alice"
  }'

curl -X POST http://localhost:6382/v1/graphs/knowledge/nodes \
  -H "Content-Type: application/json" \
  -d '{
    "label": "Project",
    "properties": {"name": "Weav", "status": "active"},
    "entity_key": "weav"
  }'

# Add an edge
curl -X POST http://localhost:6382/v1/graphs/knowledge/edges \
  -H "Content-Type: application/json" \
  -d '{"source": 1, "target": 2, "label": "WORKS_ON", "weight": 1.0}'

# Query context
curl -X POST http://localhost:6382/v1/context \
  -H "Content-Type: application/json" \
  -d '{
    "graph": "knowledge",
    "query": "what does alice work on",
    "budget": 4096,
    "max_depth": 2
  }'
from weav import WeavClient

with WeavClient() as client:
    # Create a graph
    client.create_graph("knowledge")

    # Add nodes
    alice = client.add_node(
        "knowledge", "Person",
        properties={"name": "Alice", "role": "Engineer"},
        entity_key="alice",
    )
    weav = client.add_node(
        "knowledge", "Project",
        properties={"name": "Weav", "status": "active"},
        entity_key="weav",
    )

    # Add an edge
    client.add_edge(
        "knowledge", alice, weav,
        "WORKS_ON", weight=1.0,
    )

    # Query context
    result = client.context(
        "knowledge",
        query="what does alice work on",
        budget=4096,
    )

    # Use in an LLM prompt
    print(result.to_prompt())

Installation

Docker (recommended)

# Quick start with docker-compose
docker-compose up -d

# Or build and run directly
docker build -t weav .
docker run -p 6380:6380 -p 6381:6381 -p 6382:6382 weav

The Docker image uses a multi-stage build with Rust 1.85 and produces a minimal Debian runtime image. Persistence is enabled by default with data stored in /data.

From source

git clone https://github.com/SiluPanda/weav.git
cd weav
cargo build --workspace --release
./target/release/weav-server

Requires Rust 1.85+ (edition 2024).

Full build with LLM providers

cargo build -p weav-server --features full --release

Python SDK

pip install weav

Node.js SDK

npm install @weav/client

Core Concepts

Graphs

Graphs are named containers for nodes and edges. A single Weav server can host multiple independent graphs. Each graph has its own adjacency store, property store, vector index, and string interner. Graphs are created and dropped at runtime via the API.

Nodes

Nodes are the primary entities in a graph. Each node has:

  • Node ID -- a u64 identifier assigned on creation
  • Label -- a type string (e.g. "Person", "Document", "Concept")
  • Properties -- arbitrary key-value pairs
  • Embedding -- an optional float vector for similarity search
  • Entity Key -- an optional unique string for deduplication and lookup

Edges

Edges are directed connections between two nodes. Each edge has:

  • Edge ID -- a u64 identifier
  • Source / Target -- the node IDs it connects
  • Label -- a relationship type string (e.g. "WORKS_ON", "KNOWS")
  • Weight -- a f32 value (default 1.0) used in traversal scoring
  • Properties -- optional key-value pairs

Property Values

Properties support dynamic typed values:

TypeDescription
NullAbsence of value
Booltrue / false
Int64-bit signed integer (i64)
Float64-bit floating point (f64)
StringUTF-8 text
BytesRaw binary data
VectorArray of f32 values
ListOrdered array of values
MapNested key-value object
TimestampMilliseconds since epoch (u64)

Bi-Temporal Tracking

Every entity (node and edge) in Weav carries two time dimensions:

  • Valid time (valid_from / valid_until) -- when the fact was true in the real world
  • Transaction time (tx_from / tx_until) -- when the fact was recorded in the database

This enables point-in-time queries: "What did the graph look like at timestamp T?" Entities are never physically deleted -- they are invalidated by setting valid_until, which preserves the full history for auditing and temporal queries.

Provenance

Provenance tracks the origin and confidence of every fact in the graph:

  • source -- free-text origin identifier
  • confidence -- a float from 0.0 to 1.0
  • extraction_method -- one of: LlmExtracted, NlpPipeline, UserProvided, Derived, Imported

String Interning

Labels and property keys are stored internally as LabelId / PropertyKeyId (both u16) via a StringInterner. This avoids heap-allocating repeated strings and reduces memory usage significantly for graphs with many nodes sharing the same labels and property keys.

Token Budget

Weav's context retrieval uses a greedy knapsack algorithm to pack the most valuable context chunks within an LLM's token limit. Each chunk has a relevance score and a token count; the budget enforcer selects the optimal subset by value-density (relevance per token). This is the core differentiator from plain graph or vector databases.

HTTP API Reference

All endpoints return JSON with the structure {"success": bool, "data": ..., "error": ...}. The HTTP API listens on port 6382 by default. Request bodies are limited to 10 MB. Requests time out after 30 seconds.

Health and Info

GET /health

Health check endpoint. Returns {"status": "ok"}.

curl http://localhost:6382/health
GET /v1/info

Server information including version and graph count.

curl http://localhost:6382/v1/info
GET /metrics

Prometheus metrics endpoint. Returns metrics in Prometheus text exposition format. Requires the observability feature.

curl http://localhost:6382/metrics

Graph Management

POST /v1/graphs -- Create graph

Create a new named graph.

# Request
{"name": "my_graph"}

# Response
{"success": true, "data": null}

List all graph names.

# Response

Permanently delete a graph and all its data.

Run an integrity check on a graph. Validates adjacency consistency, orphaned edges, and index state.

GET /v1/graphs/{graph}/stats/detailed -- Detailed stats

Comprehensive graph statistics: degree distribution (min/max/avg/p50/p95/p99), label distribution, edge type distribution, memory estimates, temporal node count.

Context Retrieval

The CONTEXT command is Weav's core differentiator. It retrieves graph-aware, budget-constrained context optimized for LLM consumption. No other graph database offers this: a single query that traverses the graph, scores relevance, respects token limits, and outputs LLM-ready formatted content.

Request Parameters

querystringnullText query for vector search seeding
embeddingfloat[]nullPre-computed embedding vector
seed_nodesstring[]nullEntity keys to start traversal from
budgetu324096Token budget limit
budget_presetstringnullNamed preset (overrides budget)

The 8-Stage Pipeline

A context query passes through these stages in order:

  1. Vector Search -- if a query or embedding is provided, find the nearest vectors in the HNSW index to establish seed nodes
  2. Entity Lookup -- resolve seed_nodes entity keys to node IDs via the entity key index
  3. Graph Traversal -- BFS from seed nodes up to max_depth, following edges weighted by relevance
  4. Flow Scoring -- compute relevance scores based on traversal depth, edge weight, and connectivity
  5. Temporal Filter -- if temporal_at is set, exclude entities not valid at that timestamp
  6. Relevance Decay -- apply time-based decay function to reduce scores of older content
  7. Token Budget -- greedy knapsack packing: select chunks by value-density (relevance / tokens) within the budget
  8. Format -- serialize chunks into the requested output format (raw, anthropic, openai)

Seed Strategies

StrategyWhen Used

Cypher Queries

Weav supports a subset of Cypher for pattern-matching graph queries. Submit Cypher queries via the HTTP API:

curl -X POST http://localhost:6382/v1/graphs/knowledge/cypher \
  -H "Content-Type: application/json" \
  -d '{"query": "MATCH (n:Person)-[r:KNOWS]->(m:Person) RETURN n.name, m.name, r.since LIMIT 10"}'

Supported Clauses

  • MATCH -- pattern matching with node labels and relationship types
  • WHERE -- property filters with comparison operators (=, !=, <, >, <=, >=, IS NULL, IS NOT NULL)
  • RETURN -- specify output columns, supports count() and collect() aggregations
  • LIMIT -- restrict result count
  • ORDER BY -- sort results (ASC/DESC)
  • CREATE -- create nodes with labels and properties

For deleting nodes/edges, updating properties, and shortest path queries, use the dedicated HTTP endpoints (DELETE /v1/graphs/{graph}/nodes/{id}, PUT /v1/graphs/{graph}/nodes/{id}, and POST /v1/graphs/{graph}/algorithms/shortest_path).

Examples

-- Find all people who know each other
MATCH (a:Person)-[:KNOWS]->(b:Person) RETURN a.name, b.name

-- Create a node
CREATE (n:Person {name: "Alice", role: "Engineer"})

-- Property filter with ordering
MATCH (n:Person) WHERE n.age > 25 RETURN n.name, n.age ORDER BY n.age DESC LIMIT 10

-- Count nodes by label
MATCH (n:Person) RETURN count(n)

Persistence

Weav is an in-memory database with optional persistence via write-ahead log (WAL) and snapshots. Persistence is disabled by default.

Write-Ahead Log (WAL)

  • Each WAL entry is length-prefixed, bincode-serialized, with a CRC32 checksum for corruption detection
  • The WAL records 10 operation types: graph create/drop, node add/update/delete, edge add/delete/invalidate, vector update, and document ingest
  • Maximum WAL file size is configurable (default 256 MB); WAL compaction removes entries already covered by the latest snapshot

Sync Modes

ModeDescription
Alwaysfsync after every write -- maximum durability, higher latency
EverySecondfsync once per second -- good balance (default)
NeverOS-managed sync -- fastest, risk of data loss on crash

Snapshots

  • Full-state bincode serialization of all graphs
  • Each snapshot includes a .meta.json sidecar with metadata (timestamp, graph list, entry counts, sequence number)
  • Configurable auto-snapshot interval (default: 3600 seconds)
  • Manual trigger via POST /v1/snapshot

Recovery

On startup with persistence enabled, Weav:

  1. Loads the latest valid snapshot from data_dir
  2. Replays all WAL entries with sequence numbers after the snapshot
  3. Validates CRC32 checksums; corrupted entries are skipped with a warning

Commands

EndpointDescription
POST /v1/snapshotTrigger immediate snapshot
POST /v1/backupSnapshot + WAL compact in one call
POST /v1/wal/compactCompact WAL (remove entries before latest snapshot)

Entity Resolution

Weav has built-in entity deduplication that prevents duplicate nodes when semantically equivalent entities are added. Three matching tiers are applied in priority order.

Matching Tiers

TierMethodWhenFalse Positives
1Exact entity keyAlways, when entity_key is providedNone
2Jaro-Winkler fuzzy nameWhen name_field is configured (threshold default: 0.85)Possible
3Vector embedding similarityWhen embedding provided (threshold default: 0.92)Possible

When a match is found, properties from the new node are merged into the existing node using LastWriteWins policy, and the existing node ID is returned.

Entity Key Dedup

The most common approach. Provide an entity_key when adding a node. If a node with the same key already exists, Weav merges properties instead of creating a duplicate.

# First add creates the node
curl -X POST http://localhost:6382/v1/graphs/knowledge/nodes \
  -d '{"label": "Person", "properties": {"name": "Alice"}, "entity_key": "alice@acme.com"}'
# {"success": true, "data": {"node_id": 1}}

# Second add with same key merges into existing node
curl -X POST http://localhost:6382/v1/graphs/knowledge/nodes \
  -d '{"label": "Person", "properties": {"role": "Engineer"}, "entity_key": "alice@acme.com"}'
# {"success": true, "data": {"node_id": 1}}  -- same ID, properties merged

Fuzzy Name Matching

Uses Jaro-Winkler string similarity with a character trigram blocking index for efficient candidate narrowing. Handles typos and minor name variations (e.g., "Alice Johnson" matches "Alice Jonson" at ~0.9 similarity). Requires require_same_label to also match node labels (default: true).

Node Merge Command

Explicitly merge two nodes when you know they represent the same entity.

# RESP3
NODE MERGE "knowledge" 10 INTO 20 POLICY keep_target

# Python SDK
client.merge_nodes("knowledge", source_id=10, target_id=20, conflict_policy="merge")

# Node.js SDK
await client.mergeNodes("knowledge", 10, 20, "keep_source")

The merge process: copies non-system properties from source to target (applying the conflict policy), re-links all edges from source to target, copies the embedding vector if target lacks one, then deletes the source node.

Conflict Policies

PolicyOn ConflictDescription
keep_targetKeep existingDefault. Target node's property value is preserved.
keep_sourceOverwriteSource node's property value wins.
mergeOverwrite + auditSource value wins, but conflicts are recorded in the response for review.

Skip Dedup on Ingest

The ingestion pipeline runs entity dedup by default. For bulk imports where you know entities are unique, skip it for better performance:

curl -X POST http://localhost:6382/v1/graphs/knowledge/ingest \
  -d '{"content": "...", "skip_dedup": true}'

Multi-tenancy

Weav supports logical multi-tenancy via the X-Group-Id HTTP header. When set, the group identifier is automatically stored as a _group_id property on created nodes, enabling tenant-aware data organization within a shared graph.

How It Works

Include the X-Group-Id header on node creation requests. Weav attaches it as a system property:

# Nodes created with tenant tagging
curl -X POST http://localhost:6382/v1/graphs/shared/nodes \
  -H "X-Group-Id: tenant-abc" \
  -d '{"label": "Document", "properties": {"title": "Q4 Report"}}'

# The node will have _group_id: "tenant-abc" in its properties

Querying by Tenant

Use property search to filter nodes by tenant:

# Find all nodes belonging to a tenant
curl "http://localhost:6382/v1/graphs/shared/search?key=_group_id&value=tenant-abc"

Usage Example

from weav import WeavClient

with WeavClient() as client:
    # Build a knowledge graph
    client.create_graph("research")

    paper_id = client.add_node("research", "Paper", {
        "title": "Attention Is All You Need",
        "year": 2017,
    }, entity_key="attention_paper")

    author_id = client.add_node("research", "Person", {
        "name": "Ashish Vaswani",
    }, entity_key="vaswani")

    client.add_edge("research", author_id, paper_id, "AUTHORED")

    # Query context for an LLM
    result = client.context(
        "research",
        query="transformer architecture",
        budget=4096,
        max_depth=2,
    )
    print(f"Included {result.nodes_included} nodes, {result.total_tokens} tokens")
    print(result.to_prompt())

    # Run an algorithm
    pr = client.run_algorithm("research", "pagerank", damping=0.85)
    print(pr)

Usage Example

import { WeavClient } from '@weav/client';

const client = new WeavClient();

// Create graph and add data
await client.createGraph('knowledge');

const alice = await client.addNode('knowledge', {
  label: 'Person',
  properties: { name: 'Alice', role: 'Engineer' },
  entityKey: 'alice',
});

const project = await client.addNode('knowledge', {
  label: 'Project',
  properties: { name: 'Weav' },
  entityKey: 'weav',
});

await client.addEdge('knowledge', {
  source: alice,
  target: project,
  label: 'WORKS_ON',
});

// Query context
const result = await client.context({
  graph: 'knowledge',
  query: 'what does alice work on',
  budget: 4096,
  maxDepth: 3,
});

console.log(`Included ${result.nodesIncluded} nodes`);
console.log(`Total tokens: ${result.totalTokens}`);

for (const chunk of result.chunks) {
  console.log(`[${chunk.label}] ${chunk.content} (score: ${chunk.relevanceScore})`);
}
